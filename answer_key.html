Night Mode Switcher
EASY
Complete Solution

Inputs & Outputs

Input A: Daytime sensor (1 = Day, 0 = Night)

Output: Monsters active (1 = Active, 0 = Sleep)

Logic Gate Used

NOT Gate (Inverter)

The NOT gate inverts the daytime signal so monsters appear at night only.

Truth Table

Daytime (Input A) Monsters Active (Output) Game Behavior
0 (Night) 1 (Active) Monsters appear and roam
1 (Day) 0 (Sleep) Monsters are sleeping/hidden

Circuit Flow

Daytime Sensor ‚Üí NOT Gate ‚Üí Monster Controller

When sensor reads 1 (daytime), NOT gate outputs 0 (monsters off)

When sensor reads 0 (nighttime), NOT gate outputs 1 (monsters on)

Pseudocode: READ daytime FROM sensor monsters_active = NOT daytime OUTPUT monsters_active TO monster_controller // MakeCode Block Example forever: if (daytime == 1): monsters_active = 0 else: monsters_active = 1

üí° Teaching Tips

  • Connect to real nightlights: They turn on when it's NOT bright
  • Students often forget that NOT inverts the entire logic
  • Have students verbalize: "If it IS day, monsters are NOT active"
  • Good starter challenge - builds confidence with simple inversion

‚ö†Ô∏è Common Student Errors

  • Forgetting to invert - using daytime signal directly
  • Confusing input labels (thinking 1=night instead of 1=day)
  • Drawing circuit without the NOT gate symbol
Multi-Player Door Unlock
EASY
Complete Solution

Inputs & Outputs

Input A: Player 1 pressure plate (1 = Pressed, 0 = Not pressed)

Input B: Player 2 pressure plate (1 = Pressed, 0 = Not pressed)

Output: Door unlocked (1 = Open, 0 = Locked)

Logic Gate Used

OR Gate

The OR gate allows ANY player to open the door - cooperative gameplay.

Truth Table

Player 1 (A) Player 2 (B) Door Open (Output) Game Behavior
0 0 0 Door stays locked
0 1 1 Player 2 opens door
1 0 1 Player 1 opens door
1 1 1 Both players can enter together

Circuit Flow

Player 1 Plate ‚Üí OR Gate ‚Üê Player 2 Plate ‚Üí Door Lock

Both inputs connect to the OR gate, which outputs to the door mechanism

Pseudocode: READ player1 FROM pin0 READ player2 FROM pin1 door_open = player1 OR player2 OUTPUT door_open TO door_lock // MakeCode Block Example forever: if (player1 == 1 OR player2 == 1): door_open = 1 else: door_open = 0

üí° Teaching Tips

  • Emphasize "at least one" concept for OR gates
  • Connect to real life: automatic doors open if ANYONE approaches
  • Good for discussing teamwork in games
  • Ask: "Why would game designers use OR instead of AND here?"

‚ö†Ô∏è Common Student Errors

  • Confusing OR with XOR (thinking both players can't work together)
  • Missing the case where both players press simultaneously
  • Using AND gate instead, requiring both players
Two-Key Treasure Chest
EASY
Complete Solution

Inputs & Outputs

Input A: Golden Key collected (1 = Yes, 0 = No)

Input B: Silver Key collected (1 = Yes, 0 = No)

Output: Chest opens (1 = Open, 0 = Locked)

Logic Gate Used

AND Gate

The AND gate requires BOTH keys - creates a challenging quest objective.

Truth Table

Golden Key (A) Silver Key (B) Chest Opens (Output) Game Behavior
0 0 0 Need both keys
0 1 0 Still need golden key
1 0 0 Still need silver key
1 1 1 Chest unlocks! Treasure obtained

Circuit Flow

Golden Key Inventory ‚Üí AND Gate ‚Üê Silver Key Inventory ‚Üí Chest Lock

Both key signals must be HIGH (1) for the chest to unlock

Pseudocode: READ golden_key FROM inventory READ silver_key FROM inventory chest_open = golden_key AND silver_key OUTPUT chest_open TO chest_lock // MakeCode Block Example forever: if (golden_key == 1 AND silver_key == 1): chest_open = 1 else: chest_open = 0

üí° Teaching Tips

  • Connect to login systems: need username AND password
  • Emphasize that only ONE combination works (both keys)
  • Discuss why game designers use AND for treasure (increases challenge)
  • Compare with OR gate door - why different gates for different purposes?

‚ö†Ô∏è Common Student Errors

  • Using OR gate, making puzzle too easy
  • Thinking only one key should work
  • Not understanding why three rows output 0
Light Switch Puzzle
EASY
Complete Solution

Inputs & Outputs

Input A: Switch 1 position (1 = Up, 0 = Down)

Input B: Switch 2 position (1 = Up, 0 = Down)

Output: Light on (1 = On, 0 = Off)

Logic Gate Used

XOR Gate (Exclusive OR)

The XOR gate creates a puzzle where switches must be DIFFERENT to activate light.

Truth Table

Switch 1 (A) Switch 2 (B) Light On (Output) Game Behavior
0 (Down) 0 (Down) 0 Both down - light off (puzzle unsolved)
0 (Down) 1 (Up) 1 Different positions - light on! ‚úì
1 (Up) 0 (Down) 1 Different positions - light on! ‚úì
1 (Up) 1 (Up) 0 Both up - light off (puzzle unsolved)

Circuit Flow

Switch 1 ‚Üí XOR Gate ‚Üê Switch 2 ‚Üí Light Bulb

XOR outputs HIGH only when inputs are DIFFERENT (opposite positions)

Pseudocode: READ switch1 FROM pin0 READ switch2 FROM pin1 light_on = switch1 XOR switch2 OUTPUT light_on TO light_bulb // MakeCode Block Example forever: if (switch1 != switch2): // different values light_on = 1 else: light_on = 0

üí° Teaching Tips

  • XOR is hardest gate for students to grasp initially
  • Key phrase: "one or the other, but NOT both"
  • Connect to real 3-way light switches in hallways
  • Show how XOR creates interesting puzzle mechanics
  • Demonstrate: flipping either switch toggles the light

‚ö†Ô∏è Common Student Errors

  • Confusing XOR with OR (thinking both switches can work)
  • Not understanding why matching positions = off
  • Drawing OR gate symbol instead of XOR
  • Missing the "exclusive" part of "exclusive OR"
Stealth Mission Alarm
STANDARD
Complete Solution

Inputs & Outputs

Input A: Guard spots player (1 = Spotted, 0 = Hidden)

Input B: Player has disguise (1 = Yes, 0 = No)

Output: Alarm sounds (1 = Alarm, 0 = Silent)

Logic Gates Used

Gate 1: NOT Gate - Inverts the disguise signal

Gate 2: AND Gate - Combines "spotted" with "no disguise"

Logic: Alarm = Spotted AND (NOT Disguise)

Truth Table

Spotted (A) Has Disguise (B) NOT B Alarm (Output) Game Behavior
0 0 1 0 Hidden - no alarm
0 1 0 0 Hidden with disguise - no alarm
1 0 1 1 Spotted without disguise - ALARM! üö®
1 1 0 0 Spotted but disguised - safe!

Circuit Flow (2-Stage)

Stage 1: Disguise Input ‚Üí NOT Gate ‚Üí "No Disguise" signal

Stage 2: "No Disguise" + Spotted ‚Üí AND Gate ‚Üí Alarm Output

Complete Flow: Both conditions must be true for alarm

Pseudocode: READ spotted FROM guard_sensor READ has_disguise FROM inventory no_disguise = NOT has_disguise // Gate 1 alarm = spotted AND no_disguise // Gate 2 OUTPUT alarm TO alarm_system // MakeCode Block Example forever: no_disguise = NOT has_disguise if (spotted == 1 AND no_disguise == 1): alarm = 1 else: alarm = 0

üí° Teaching Tips

  • This is students' first multi-gate challenge - emphasize gate ORDER
  • Have students identify which gate comes FIRST in the logic
  • Draw intermediate signal on circuit diagram (label "no_disguise")
  • Discuss: disguise acts as "protection" - negates the spotted condition
  • Compare to real security: badge + face scan (both needed)

‚ö†Ô∏è Common Student Errors

  • Forgetting to invert disguise signal first
  • Using OR instead of AND (alarm would sound too easily)
  • Connecting gates in wrong order
  • Not labeling the intermediate "no disguise" signal
  • Missing truth table row where both inputs are 1
Power-Up Combo System
STANDARD
Complete Solution

Inputs & Outputs

Input A: Full health (1 = Full, 0 = Not full)

Input B: Full energy (1 = Full, 0 = Not full)

Input C: Button pressed (1 = Pressed, 0 = Not pressed)

Output: Super move activates (1 = Activate, 0 = No move)

Logic Gates Used

Gate 1: OR Gate - Checks if health OR energy is full

Gate 2: AND Gate - Combines "powered up" with button press

Logic: Super Move = (Health OR Energy) AND Button

Truth Table

Health (A) Energy (B) Button (C) A OR B Super Move (Output)
00000
00100 - Not powered up!
01010 - Didn't press button
01111 - SUCCESS! ‚ö°
10010 - Didn't press button
10111 - SUCCESS! ‚ö°
11010 - Didn't press button
11111 - SUCCESS! ‚ö°

Circuit Flow (2-Stage)

Stage 1: Health + Energy ‚Üí OR Gate ‚Üí "Powered Up" signal

Stage 2: "Powered Up" + Button ‚Üí AND Gate ‚Üí Super Move

Key Point: Player must be powered up (either way) AND press button

Pseudocode: READ full_health FROM player_stats READ full_energy FROM player_stats READ button_pressed FROM controller powered_up = full_health OR full_energy // Gate 1 super_move = powered_up AND button_pressed // Gate 2 OUTPUT super_move TO game_engine // MakeCode Block Example forever: powered_up = (full_health OR full_energy) if (powered_up AND button_pressed): activate_super_move()

üí° Teaching Tips

  • This introduces 3-input problems (good progression)
  • Emphasize hierarchy: OR happens first, then AND
  • Connect to real games (fighting games, RPGs with special moves)
  • Truth table has 8 rows - walk through systematically
  • Ask: "Why would designers require BOTH conditions?"

‚ö†Ô∏è Common Student Errors

  • Using AND for health/energy (would require both - too restrictive)
  • Missing intermediate "powered up" signal in diagram
  • Forgetting to test all 8 input combinations
  • Reversing gate order (AND before OR)
Boss Battle Weakness System
ADVANCED
Complete Solution

Inputs & Outputs

Input A: Fire weapon equipped (1 = Yes, 0 = No)

Input B: Ice weapon equipped (1 = Yes, 0 = No)

Input C: Boss shield down (1 = Down, 0 = Up)

Input D: Raining (1 = Yes, 0 = No)

Output: Boss takes damage (1 = Damage, 0 = No damage)

Logic Gates Used (3-Gate Design)

Gate 1: OR Gate - Checks for any special weapon

Gate 2: NOT Gate - Inverts rain signal (not raining)

Gate 3: AND Gate - Combines weapon + shield + weather

Logic: Damage = (Fire OR Ice) AND Shield_Down AND (NOT Raining)

Truth Table (Key Combinations)

Fire Ice Shield Rain A OR B NOT D Boss Damage
0010010 - No weapon
1000110 - Shield blocks
1011100 - Rain extinguishes
1010111 - SUCCESS! üî•
0110111 - SUCCESS! ‚ùÑÔ∏è
1110111 - SUCCESS! üî•‚ùÑÔ∏è

Circuit Flow (3-Stage)

Stage 1a: Fire + Ice ‚Üí OR Gate ‚Üí "Has Weapon"

Stage 1b: Rain ‚Üí NOT Gate ‚Üí "Not Raining"

Stage 2: "Has Weapon" + Shield_Down + "Not Raining" ‚Üí AND Gate ‚Üí Damage

All 3 conditions must be TRUE simultaneously!

Pseudocode: READ fire_weapon FROM inventory READ ice_weapon FROM inventory READ shield_down FROM boss_status READ raining FROM weather has_weapon = fire_weapon OR ice_weapon // Gate 1 not_raining = NOT raining // Gate 2 damage = has_weapon AND shield_down AND not_raining // Gate 3 OUTPUT damage TO boss_health // MakeCode Block Example forever: has_weapon = (fire_weapon OR ice_weapon) not_raining = NOT raining if (has_weapon AND shield_down AND not_raining): boss_takes_damage()

üí° Teaching Tips

  • This is first 4-input challenge - expect students to need support
  • Break down into stages: weapon check ‚Üí weather check ‚Üí combine all
  • Emphasize that OR and NOT happen in parallel, then feed into AND
  • Truth table has 16 rows - students can show selective rows
  • Connect to real boss battles in games (multi-condition weaknesses)
  • Discuss game balance: why make boss fights complex?

‚ö†Ô∏è Common Student Errors

  • Forgetting to invert the rain signal
  • Using AND for weapons (would require both weapons)
  • Not showing intermediate signals in circuit diagram
  • Confusion about which gates operate in parallel vs. series
  • Incomplete truth table (missing combinations)

üìö General Teaching Notes

üéØ Assessment Guidelines

  • Focus on Understanding: Can student explain WHY they chose each gate?
  • Truth Table Accuracy: This is the most objective measure of correctness
  • Circuit Clarity: Can another student understand their diagram?
  • Documentation Quality: Are labels clear and complete?
  • Real-World Connection: Can they relate logic to actual games/devices?

‚è±Ô∏è Time Expectations

  • EASY Challenges: 20-30 minutes (1 gate)
  • STANDARD Challenges: 35-45 minutes (2 gates)
  • ADVANCED Challenges: 50-60 minutes (3 gates, 4 inputs)
  • Physical Model Building: Add 30-45 minutes
  • Presentations: 5 minutes per student/group

üîç Look-Fors During Assessment

  • Correct Gate Selection: Did they choose appropriate gates for the logic?
  • Proper Gate Ordering: For multi-gate designs, is the sequence correct?
  • Complete Truth Tables: All input combinations tested?
  • Labeled Diagrams: All inputs, outputs, and gates clearly marked?
  • Intermediate Signals: For multi-gate, are connections between gates shown?
  • Game Logic Accuracy: Does the PLD actually solve the game scenario?
  • Thoughtful Reflection: Can they articulate their design choices?

üí¨ Discussion Questions for Class

  • Why would game designers choose OR vs AND for different mechanics?
  • How does combining gates create more complex behaviors?
  • What real-world devices use similar logic (security systems, appliances)?
  • How do logic gates relate to the "if-then" statements in coding?
  • Why is it important to test ALL input combinations?
  • How could you modify your design to make it more challenging/easier?

üõ†Ô∏è Differentiation Strategies

  • For Struggling Students:
    • Start with EASY challenges only
    • Provide partially completed truth tables
    • Use concrete manipulatives (colored cards for 0/1)
    • Partner with stronger student
    • Allow use of TinkerCAD simulator
  • For Advanced Students:
    • Challenge them with ADVANCED cards immediately
    • Ask them to create their own game scenarios
    • Have them build physical models with micro:bit
    • Introduce NAND/NOR gates as extension
    • Have them write actual micro:bit code

‚úÖ Quick Grading Tips

  • Truth Table: Check one "success" row and one "failure" row for correctness
  • Circuit Diagram: Verify gates match the challenge requirements
  • Labels: Spot-check that inputs and outputs are clearly marked
  • Reflection Quality: Look for specific examples, not generic responses
  • Use Student Self-Assessment: Have students grade themselves first
  • Focus Feedback: Pick 1-2 areas for improvement rather than marking everything
